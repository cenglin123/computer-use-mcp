# 系统设计概览

> 记录系统级设计决策和架构选型理由。只写“为什么这样设计”，不写代码能直接看到的“是什么”。
> 修改本文件后同步更新 [CHANGELOG.md](../CHANGELOG.md)。

## 系统定位

`computer-use` 是一个本地 MCP 服务器，让大模型 Agent 能够“看到”屏幕并“操作”Windows GUI。它同时提供一个本地调试 CLI，用于开发和手动验证。

## 架构主线

MCP 客户端（如 Claude Desktop、Codex、Kimi Code 或其他支持 stdio MCP 的客户端）调用本服务器的 tools → `mcp_server.py` 解析请求 → `core.py` 执行截图/鼠标/键盘/滚动操作，**截图保存为本地 PNG 文件并只返回文件路径引用**，模型通过客户端提供的多模态文件读取能力按需读取图像 → 返回操作结果。

`ui_automation.py` 作为可选补充，用于在坐标点击之外直接获取控件信息，提高稳定性。`snapshot.py` 在 UIA 可用时生成结构化控件树快照，让模型可以凭 UID 操作控件，跳过反复截图估算坐标。`trace.py` 记录每次调用的步骤、结果、耗时和错误，支持事后复盘。`safety.py` 为公开执行入口提供主屏坐标、目标窗口和输入内容检查；`core.py` 在最终输入原语再次强制主屏坐标边界，避免直接调用绕过。

关键不变量：所有坐标都使用**物理虚拟屏幕像素**（mss 坐标系），与截图像素 1:1 对应；只读感知可覆盖虚拟桌面和副屏，真实输入只允许主屏内非负坐标；不允许在安全检查失败时静默降级执行。

## 关键设计决策

### 为什么坐标系与 mss 截图像素 1:1 绑定

- **场景**：多显示器、DPI 缩放、虚拟桌面混合时，逻辑坐标和物理坐标容易错位，导致 Agent 看到的位置和点击的位置不一致。
- **备选**：让 Agent 使用系统逻辑坐标，再在内部转换。
- **选择**：强制所有工具接口使用物理虚拟屏幕像素。`CoordinateSystem` 在启动时读取 mss 的 monitor 配置，并拒绝混合 DPI 的显示器组合。
- **取舍**：Agent 看到的坐标数字会大于常见逻辑坐标，但消除了“截图对得上却点不准”的最大一类错误。

### 为什么感知跨屏而输入只限主屏

- **场景**：模型需要看到完整虚拟桌面来理解窗口位置，但跨屏真实输入会扩大远控、多 DPI 和负坐标环境下的误操作范围。
- **选择**：截图、显示器枚举、控件检查和 UI 快照保留跨屏能力；鼠标、滚动和键盘输入只接受主屏内非负坐标。主屏规则在最终输入原语强制，公开入口再叠加目标窗口和内容检查。
- **取舍**：副屏只能观察、不能操作，牺牲部分多屏自动化能力，换取明确且不可绕过的输入边界。

### 为什么坐标检查与目标检查分层

- **场景**：`core.py` 是所有 PyAutoGUI 输入的最低执行层，但目标窗口检查依赖可选 UIA；把 UIA 引入 core 会形成不必要的耦合和循环依赖。
- **选择**：core 只承担不可绕过的主屏坐标检查；MCP、CLI、复合工具和快照点击在执行前按实际坐标完成目标窗口与文本检查。快照携带的进程或窗口元数据不作为安全事实。
- **取舍**：入口层存在一定重复校验，但保留了纵深防御，并使无 UIA 环境仍能使用低层输入原语。

### 为什么优先用 pyautogui 而不是 SendMessage

- **场景**：需要跨应用点击、滚动、输入文本，目标应用没有公开 API。
- **备选**：对目标窗口句柄发送 `WM_LBUTTONDOWN` / `WM_CHAR` 等消息。
- **选择**：使用 `pyautogui`，它通过 Windows `SendInput` 发送真实输入事件。
- **取舍**：会和用户共享全局光标/键盘，无法并行操作；但兼容性最好，能处理自定义绘制控件、浏览器、UWP 等难以用消息驱动的场景。

### 为什么保留 UI Automation 作为可选路径

- **场景**：部分 Windows 应用支持 UIA，能直接获取控件名称、矩形、进程信息。
- **选择**：`ui_automation.py` 只在安装了 `uiautomation` 包时启用；失败时静默降级到坐标操作。
- **取舍**：增加了一个可选依赖，但能显著提升“查找按钮”类任务的稳定性，且不影响无 UIA 环境。

### 为什么 MCP 只保留最简键鼠宏 + 观察能力

- **场景**：用户一句话触发 GUI 操作，模型需要拆解成键鼠宏并执行；复杂多步任务需要模型边观察屏幕边 ReAct 迭代，而不是靠 MCP 内置复杂编排。
- **选择**：MCP 只提供原语（点击、移动、拖拽、按键、滚动、截图、控件查找、应用启动）和安全检查；多步规划、循环、条件判断、重试由上层 Agent 框架用 ReAct 完成。能用命令行更简洁完成的任务（如批量文件处理、系统命令）交给 Bash，不塞进 MCP。
- **取舍**：MCP 自身不实现工作流引擎或状态机，保持轻量；Agent 需要多轮交互，但每一轮都能基于最新截图动态调整，避免把“判断”硬编码进工具。

### 为什么截图不返回 base64，只返回文件路径

- **场景**：长 GUI 任务会累积大量截图，base64 会快速撑满模型上下文并拖慢每次请求。
- **选择**：`screenshot` 工具把 PNG 保存到本地磁盘，只在响应里返回 `saved_path` 引用；多模态模型通过 `ReadMediaFile` 按需读取。
- **取舍**：需要一轮额外工具调用来读图，但把图像数据从 MCP 调用和上下文里剥离，显著降低上下文压力。

### 为什么引入 guidance 单一事实源

- **场景**：公开分发后，不同 MCP 客户端对 prompts、Skill、文档和工具描述的支持不一致；如果每层各写一套使用纪律，容易漂移。
- **选择**：`computer_use.guidance` 保存核心 agent 使用准则，MCP prompts、doctor、Skill 和通用文档围绕它派生或引用。安装后优先通过 MCP prompt `computer_use_guidance` 暴露给客户端。
- **取舍**：仍需要维护 README 和文档中的上下文说明，但核心能力边界和标准执行闭环有了单一事实源。

### 为什么截图叠加当前光标标记

- **场景**：自定义绘制界面缺少可靠 UIA 控件时，需要根据截图复盘实际光标落点，区分“坐标判断错误”和“控件未响应”。
- **选择**：所有截图在保存前于当前光标位置绘制红色十字和中心圆点；单显示器截图按显示器偏移换算，光标不在捕获范围内时不绘制。
- **取舍**：截图不再是原始屏幕像素的无损记录，标记可能遮挡少量内容；换取了 GUI 点击定位和远控 fail-safe 排查所需的明确反馈。

### 为什么引入结构化 UI 快照

- **场景**：模型每做一次动作都要先看图再估算坐标，往返次数多；chrome-devtools 的 accessibility tree + UID 模式能显著减少决策回合。
- **选择**：新增 `get_ui_snapshot`，基于 `uiautomation` 返回带快照级 UID 的控件树摘要。`click_by_uid` 等工具（阶段 2）凭 UID 操作，失败时返回 `stale_uid`。
- **取舍**：UID 仅对当前快照有效，UI 变化后必须重新取快照；但比反复截图估算坐标更稳定、更高效。

### 为什么添加结构化 Trace

- **场景**：GUI 任务出错后难以复现，缺少按任务组织的执行轨迹。
- **选择**：每次工具调用都写入 `~/.computer-use/traces/<trace_id>/trace.jsonl`，包含步骤、参数、结果、耗时、截图/快照路径和错误类型。`batch` 返回 `trace_id` 供复盘使用。
- **取舍**：增加了磁盘写入；但为错误分析、性能审计和持续优化提供了可沉淀的数据。

### 为什么响应必须携带 trace 证据

- **场景**：执行侧只拿到 `trace_id` 时，容易把目录名、空目录或历史截图误当成本次任务证据，进而向用户汇报不存在的快照或截图。
- **选择**：batch、run_task_plan 和 review_task 的响应由 trace 层 manifest 派生 `trace_path`、`artifact_root` 和 `artifacts`。manifest 只列真实存在的文件，截图 PNG 和 UI-tree JSON 在 trace 上下文内按类型分目录。
- **取舍**：每次响应会增加少量文件系统读取；换取执行侧不需要猜测目录语义，也能在复盘时直接定位可验证证据。

### 为什么引入业务 task 会话

- **场景**：同一天内 Agent 可能连续执行多个业务任务，单条 trace 只能说明一次顶层调用，不能可靠回答“这些调用属于哪个用户任务”。
- **选择**：在 trace 之上新增显式 `task_id`，形成 `task -> traces -> artifacts` 的两级审计模型。调用方通过 `start_task` 获取业务任务 ID，并在后续顶层调用中显式传入；未传时只创建一次性 standalone task 兼容旧调用方。
- **取舍**：调用方需要维护一个业务任务 ID，但避免了用时间相邻、目录位置或 Agent 对话 ID 推断归属，服务端也不需要维护易串任务的进程全局“当前任务”。

### 为什么 trace 和 task 使用日期分区加 locator

- **场景**：长期运行后所有 trace 平铺在根目录不利于浏览和归档，自定义 ID 也不能依靠名称中的日期推导路径。
- **选择**：新 trace 与 task 按创建时本地系统日期写入 `YYYY/MM/DD` 分区，使用基于原始 ID 的哈希 locator 精确定位；locator 是可重建派生数据，事实源仍是分区目录内的 JSON。
- **取舍**：读取时多一次 locator 解析；换取跨日任务、自定义 ID、旧扁平布局兼容和索引重建能力。

### 为什么任务级执行仍不引入 LLM

- **场景**：用户希望“一句话完成任务”，但 server 内部如果引入 LLM 就会变成第二个推理层，与上层 Agent 的职责冲突。
- **选择**：`run_task_plan` 只接受结构化步骤（`{"tool": ..., "args": ...}`），server 内部确定性执行并记录 trace；所有自然语言意图解析和多步决策仍由上层 Agent / 多模态模型完成。
- **取舍**：server 保持轻量、可预测、可审计；Agent 需要把高层目标拆成工具步骤，但这是 MCP 工具链应有的边界。

### 为什么复合工具只依赖 UIA 并在失败时返回结构化错误

- **场景**：常见 GUI 模式（打开菜单、填表、滚动查找）若每一步都走一次 MCP 调用，往返开销大。
- **选择**：`open_menu`、`fill_form`、`scroll_until` 等复合工具在 server 内部串行执行原子操作，失败时返回 `ui_not_found` 和候选控件，不替模型做视觉判断。
- **取舍**：自定义绘制控件或 UIA 覆盖不足的场景仍会失败，但错误信息足够上层多模态模型回退到截图 + 坐标点击；server 不引入 OCR 或 LLM。
